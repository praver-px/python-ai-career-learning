# 轮子复用与源码阅读规范

这份文档解决一个真实工作问题：AI 如果没有被明确要求，经常会重新写一套已有的通用方法，导致重复造轮子。学习时也容易走偏：要么什么都自己写，效率低；要么全靠库，原理没学到。

正确目标是：

```text
工作中优先复用成熟轮子，学习中保留最小手写练习理解原理。
```

---

## 1. 开发前必须先问的 5 个问题

每次写代码前，先检查：

1. Python 标准库有没有？
2. Django / DRF / Scrapy / Playwright 有没有内置能力？
3. 公司项目里有没有已经封装好的工具函数、基类、中间件、装饰器？
4. 是否有成熟第三方库可以直接用？
5. 如果要自己写，是为了学习原理，还是因为现有方案确实不满足？

如果答案是“已经有成熟轮子”，默认先用已有方案。

---

## 2. 优先级顺序

推荐优先级：

```text
Python 标准库
  -> 框架内置能力
  -> 公司已有封装
  -> 成熟第三方库
  -> 自己封装薄包装
  -> 自己从零实现
```

不要一上来就写工具函数。很多日常需求标准库已经有：

```text
路径处理：pathlib
JSON：json
CSV：csv
时间：datetime / zoneinfo
日志：logging
命令行参数：argparse
配置：configparser / os.environ
哈希：hashlib / hmac
URL 处理：urllib.parse
并发：concurrent.futures / asyncio
数据库 SQLite：sqlite3
```

Django 里也不要重复造：

```text
路由：urls
配置：settings
ORM：models / QuerySet
后台：Admin
分页：Paginator 或 DRF pagination
权限：auth / permissions
中间件：Middleware
缓存：cache framework
表单校验：forms / serializers
管理命令：management commands
```

爬虫里也要优先复用：

```text
HTTP 请求：requests / httpx
HTML 解析：BeautifulSoup / lxml
工程化爬虫：Scrapy
浏览器自动化：Playwright
验证码实验：ddddocr
重试：tenacity 或框架内置 retry
任务调度：Celery / APScheduler
```

## 3. 学习和工作要分开

## 学习模式

可以自己写最小版本，目的是理解原理。

例子：

```text
自己写一个简单 retry 装饰器，理解重试和异常。
自己写一个小型 CSV 保存函数，理解文件和编码。
自己写一个本地 sign 签名函数，理解 timestamp、nonce、hash。
```

要求：

- 代码量要小。
- 只覆盖核心原理。
- 写完后必须对比成熟库或框架能力。

## 工作模式

优先使用成熟轮子。

例子：

```text
项目里不要手写复杂分页，优先用 Django/DRF 分页。
不要自己写一套日志系统，优先用 logging 和项目现有日志配置。
不要自己写复杂重试框架，优先用 Scrapy retry middleware、tenacity 或项目封装。
不要自己拼 SQL，优先用 ORM 或参数化查询。
```

要求：

- 先查现有能力。
- 写清选型理由。
- 只在必要时做薄封装。

## 4. AI 使用时必须加的约束

以后让 AI 写代码时，提示词里必须加这段：

```text
在给出实现前，请先检查 Python 标准库、当前框架内置能力、常见成熟第三方库是否已经能解决，不要重复造轮子。
如果已有成熟方案，请优先使用成熟方案，并说明为什么。
如果为了学习需要手写最小实现，请明确标注“这是学习版，不建议生产直接使用”。
```

更完整的提示词：

```text
你是一个资深 Python/Django/爬虫工程师。请基于我的需求给出实现方案。

要求：
1. 先列出可复用的标准库、框架能力、公司封装或第三方库。
2. 判断是否需要自己写代码，避免重复造轮子。
3. 如果使用库，请给出最小可运行示例。
4. 如果手写实现，请说明这是学习版还是生产可用版。
5. 给出测试方法和常见坑。
```

## 5. 代码评审清单

提交代码前检查：

- 是否重复实现了标准库已有功能？
- 是否重复实现了 Django/DRF/Scrapy 已有功能？
- 是否没看公司项目已有工具函数就新建了工具函数？
- 是否为了一个小需求引入了过重依赖？
- 是否把第三方库用成黑盒，没有理解关键参数？
- 是否缺少测试或运行示例？
- 是否能说清为什么不用现成方案？

如果不能说清楚，就先停下来查资料。

## 6. 常见场景选择表

| 需求 | 优先使用 | 不建议一开始就做 |
| --- | --- | --- |
| 路径拼接 | pathlib | 字符串拼接路径 |
| 日志 | logging | 自己写 print 日志系统 |
| CSV 读写 | csv / pandas | 手写 split 解析 CSV |
| JSON 处理 | json / pydantic | 手写字符串拼 JSON |
| HTTP 请求 | requests / httpx | socket 从零写 HTTP |
| HTML 解析 | BeautifulSoup / lxml | 正则硬抠复杂 HTML |
| Django API | DRF | 自己手写大量 JSON 响应封装 |
| 分页 | DRF pagination / Paginator | 每个接口重复写分页 |
| 权限 | Django auth / DRF permissions | 每个接口复制 if 判断 |
| 重试 | Scrapy retry / tenacity | 到处复制 for retry |
| 定时任务 | Celery / APScheduler | while True + sleep |
| 签名摘要 | hashlib / hmac | 自己实现 MD5/SHA |
| OCR 实验 | ddddocr | 自己训练 OCR 模型 |

## 7. 源码阅读要求

复用轮子不代表不学习。每用一个重要库，都要至少读懂：

```text
它解决什么问题
最小使用方式
核心参数
失败时怎么排查
和替代方案相比有什么取舍
```

对公司魔改 Django 框架尤其要读：

- 基类 View / Controller
- 统一响应封装
- 权限装饰器
- 中间件
- 异常处理
- 分页封装
- 查询过滤封装
- 任务调度封装
- 日志封装

阅读输出写在：

```text
notes/wheel-reading/
```

模板：

```markdown
# 轮子阅读记录：库名或公司封装名

## 它解决什么问题

## 最小使用示例

## 核心参数

## 常见错误

## 我在项目中怎么用

## 我是否需要手写学习版
```

---

## 8. 每周新增要求

从现在开始，每周复盘必须多加一节：

```markdown
## 本周复用了哪些轮子

## 哪些地方原本想自己写，后来发现已有方案

## 我为了理解原理手写了哪个最小版本

## 这个手写版本为什么不直接用于生产
```

这能同时保证两件事：

```text
不重复造轮子。
也不因为只会调用库而失去底层理解。
```
